Welcome to my leaving cert computer science project for 2023.

This project models chess in python.
I am using the external library PyGame to provide a graphical interface.
I am basing this game on the Sanskrit version of chess (from 500-700 AD) which did not have a check or checkmate system. The king could be taken like a normal piece and this would end the game.

There are a few core features I want to complete:

  • A turn system
  • Enforce correct movement of pieces
  • Pawn promotion to Queen
  • Automatic movement of pieces
  • Game save states

Since chess is a complex game with many different pieces, I needed my project architecture to be flexible enough to process this.

I decided to go for an object-oriented approach to my design. Creating two major classes in which everything works off.

The "Board" and "ChessPiece" classes effectively hold all of the game's code.

A "ChessPiece" object has a colour value, piece type value, position value and a couple of status values for taking pieces.

Meanwhile, the "Board" class houses multiple "ChessPiece" objects. It handles setting up the board, deciding whose turn it is, logging move data for analysis, saving the game and loading the game. It can also move a piece for a player automatically.

By splitting the code into these two sections, we can process a variety of pieces on the board with relative ease.

The game can process both keyboard input, mouse input and touch input.

It has been designed to accept a mouse click anywhere within the window. The program is intelligent enough to be able to reject an input if it is invalid (e.g. the user has not clicked on the ChessPiece, rather they clicked on empty space)

The same is true for keyboard input. The keyboard is not required to play the game, but it adds additional functionality that the user can take advantage of.

For example, saving the game, loading up a previous session, resetting the board and enabling single-player mode.

After completing the code for mouse input, I loaded the game onto a computer which has a touch screen. After a few adjustments to my code, the game is now fully playable on touch devices. Granted some functionality is lost on touch-only devices as some functions are bound to the keyboard.

The game can compile the current state of the board into a save file. This file details the current position of all the pieces on the board, which pieces have been taken, whose turn it is and a log history of every move made so far. The logs include who moved what piece, where they moved it and how long it took them to do so.

We can analyse these save files to extract data such as the most (max) and least (min) commonly visited squares and the average (mean) time taken per move. We can use this to create a heat map of the most frequently visited squares on the board.

Implementation

500 words This section should provide a clear description of how the system was implemented. You should explain the process you went through in creating your artefact. Git was used during the entire development of the project. This lets me keep track of progress and roll back the project to a previous state should something go catastrophically wrong. The project can be found here.

I knew that I wanted to design in an OOP style. In the main program, there are no global variables. Everything must be passed from one function or another. With this in mind, we structure our code so that a board object contains many chess piece objects. Every chess piece takes the parent board as a parameter, this way each chess piece has access to the data of every other chess piece. The same is done for the GUI window, the board class takes the screen as parameter "ctx". With this system, every object has access to every other object in a consistent structure.

Originally I planned to store all chess piece objects in a 2d array with their position in the array representing their position on the board. This created problems when rendering the pieces and checking where pieces could move to. The code would've needed to loop over the 2d array many times to retrieve coordinate data. This proved to be a clunky solution that I was not comfortable working with.

One of the largest problems I had to solve regarded the "check_position" function. This compares where a piece is to where it wants to go and returns whether or not a piece can go there. It has a few general rules that apply to all pieces, but it then switches its logic to be specific to the piece type, e.g. Knight, Bishop, etc.

Some pieces were relatively simple to implement. The Rook component checks if only one axis has changed, the Bishop component checks if the slope between the new position and previous position has an absolute value of 1, the Queen combines both of these checks and the Knight can only move in a radius of 2.2 squares away from its starting position (this was calculated using the Pythagorean theorem). Pawns have their own set of rules, this includes when they are allowed to move two squares, when they can take a piece, what direction they move when taking a piece and what direction they can move based on their colour.

However, the most tedious piece to implement was the King. Ordinarily, the King moves in a radius of 1.4 or 1.0 squares. The only exception to this is if the king wants to castle. Under these circumstances, the game must figure out which Rook the King is castling with, whether that specific King and Rook pair have a clear path between them and neither piece has moved yet. If any one of these checks fails, then the King will not castle. This required adding a new "moved" attribute to pieces to denote if they have moved yet. Additionally, this had to be represented appropriately in the game save file, this was done by adding a 0 tick move on the Rook, impossibly fast for a human, so therefore it must've castled.

Since my game handles keyboard and mouse input, it needs to be able to process a wide range of values from these sources. Notably, it needs to reject any inputs from unmapped keys, and it must be able to recover if the user clicks outside the window or on an empty space.

When the user clicks on the screen, it tries to find which piece the mouse is closest to. It does this by drawing a line from the mouse to the centre of every piece on the board. Afterwards, it selects the piece with the shortest distance to the mouse and notifies the rest of the program. Two problems arose with this: originally it did not get the distance from the centre of the pieces but rather the top right corner. This would cause issues where if you clicked a piece on its right side, you would pick up the piece to the right of the one you thought you would pick up. Additionally, the code did not know what do to if you clicked an empty space as it assumed something had to be there resulting in a crash.

During development, it was paramount that I could switch between my coordinate notation for pieces and chess board square names. Co-ordinates are tuples with an X and Y value, both starting at 0 and maxing out at 7, meanwhile, square names are two character strings with a letter (A through H) and a number (1 through 8). I created two sister functions which work as the inverse of each other. The test table below shows the code converting from square names to co-ordinates

Because of this, the function will only fail if the letter is unknown.
